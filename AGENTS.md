## Layout & install
- `install` is a symlink installer. It symlinks `nvim/`, `tmux/.tmux.conf`, `ghostty/config`, `zsh/`, and the legacy terminals into place. Changing a config's location means updating `install`.
- `nvim/` is the **live** Neovim config (LazyVim). `nvim2/` is an **archived backup** of the previous custom config ‚Äî it is NOT symlinked by `install` and should not be linked.
- Legacy terminal configs (`alacritty/`, `kitty/`, `wezterm/`, `zellij/`) are intentionally left untouched; don't refit them without being asked.

## Neovim (LazyVim)
- Standard LazyVim starter structure: `lua/config/` + `lua/plugins/`. Keep changes idiomatic LazyVim (plugin spec overrides), not bespoke structure.
- Colorscheme is **rose-pine** (set via `LazyVim` opts in `lua/plugins/colorscheme.lua`).
- File explorer is **oil.nvim**, not neo-tree (neo-tree is disabled). Open with `-` or `<leader>e`.
- PHP LSP is **intelephense** (chosen over phpactor) plus **laravel_ls** (Laravel awareness); Vue uses `vue_ls`/Volar + vtsls via the `lang.vue` + `lang.typescript` extras. Extras are imported in `lua/config/lazy.lua` and tracked in `lazyvim.json`.
- `lazy-lock.json` is gitignored. Running a headless bootstrap with the real config symlinked writes `lazy-lock.json` into `nvim/` ‚Äî harmless but stray; remove it.
- Syntax-check lua without plugins: `nvim --headless -u NONE -c "lua loadfile('<file>')" -c qa`.
- Full plugin-load check without touching the live config: copy `nvim/` into a throwaway `XDG_CONFIG_HOME` (and reuse `~/.local/share/nvim` as `XDG_DATA_HOME`), then `nvim --headless "+Lazy! load all" +qa`. Inspect merged plugin opts with `require("lazy.core.plugin").values(plug, "opts", false)`.
- lazy.nvim opts merge (`lazy/core/util.lua` `merge`): a **non-empty list replaces**, an empty table `{}` also replaces. So to clear an extra's list value (e.g. disable a linter the `lang.php` extra added) set it to `{}` in a later spec ‚Äî `linters_by_ft.php = {}` wins over the extra's `{ "phpcs" }`.

## Snacks picker ‚Äî finder/grep hidden & ignored (`nvim/lua/plugins/snacks.lua`)
- The fuzzy finder is **snacks.nvim** picker (LazyVim default). Per-source opts live under `opts.picker.sources.<source>`. We set `files = { hidden = true }` and `grep = { hidden = true, ignored = true, exclude = { "node_modules", "vendor" } }`.
- Option semantics (verified in snacks source `picker/source/files.lua` + `grep.lua`): `hidden = true` adds `--hidden` to fd/rg (shows dot-dirs) but does **not** disable gitignore, so `.gitignore`'d dirs stay out. `ignored = true` adds rg `--no-ignore` (greps gitignored files too). `exclude` is a list of rg globs (`-g !<x>`) applied on top ‚Äî that's how we keep `node_modules`/`vendor` out of grep even with `--no-ignore`.
- **`.git/` is always excluded** regardless of `hidden`/`ignored`: files source hardcodes `-E .git`(fd)/`-g !.git`(rg); grep hardcodes `--glob=!.git` (and `!.bare`). These are explicit globs, not gitignore, so `ignored=true` does not resurface `.git`.
- Design rationale: files finder keeps respecting gitignore (so `node_modules` doesn't flood the name picker), but grep intentionally searches gitignored files (owner wants generated/local files greppable), with only the bulky dirs excluded.
- Verify merged opts headlessly via the throwaway-XDG copy trick (above): `require('lazy.core.plugin').values(p,'opts',false).picker.sources`. Verify actual behavior by replaying the rg args the sources build against a fixture repo with `.claude/`, `.git/`, and a gitignored `node_modules/` + a gitignored non-vendor file.

## PHP / Laravel workflow (intelephense + php-cs-fixer)
All PHP tuning lives in `nvim/lua/plugins/lsp.lua` (overrides on top of the `lang.php` extra), except server *selection* which is a global (see first bullet).
- **Server selection is `vim.g.lazyvim_php_lsp = "intelephense"` in `lua/config/options.lua` ‚Äî THIS is what enables intelephense, not lsp.lua.** The `lang.php` extra does `local lsp = vim.g.lazyvim_php_lsp or "phpactor"` at module-load time, then enables `[lsp]` and disables the other. Default is `"phpactor"`. It MUST be set in `options.lua` (loads before specs are imported); setting it in a plugin spec is too late. **Real root cause of the "gd/gr/K do nothing" reports across PRs #2/#3:** the global was unset *and* lsp.lua disabled phpactor, so intelephense stayed `enabled=false` and NO php server attached at all ‚Äî confirmed headless (only `tailwindcss`/`laravel_ls` on the buffer, intelephense absent, empty `lsp.log`). With the global set, intelephense attaches in <5s and hover/def/refs all work (verified on ~/code/recontrol: `gr`‚Üí155 refs, `gd`‚Üíproject + `vendor/`, `K`‚Üíclass hover, clean `lsp.log`). The whole intelephense `settings` block in lsp.lua is inert unless this global makes intelephense the enabled server.
- **`gr` and `context.includeDeclaration`:** intelephense's ReferenceProvider throws `TypeError: Cannot read properties of undefined (reading 'includeDeclaration')` and aborts (‚Üí Snacks "No results found") if a `textDocument/references` request arrives without `params.context`. The current Snacks (`source/lsp/init.lua`) already sends `context = { includeDeclaration = true }` (source default `include_declaration = true`), verified by capturing the outgoing request ‚Äî so the real `gr` path is fine once intelephense runs. The TypeError only reproduces from a context-less request; it is NOT produced by current Snacks. Do not add a context shim.
- **LSP nav keymaps (gd/gr/K): do NOT override them in `servers["*"].keys`.** This LazyVim version sets `opts_extend = { "servers.*.keys" }` (`lazyvim/plugins/lsp/init.lua`), so a `servers["*"].keys` list is *appended* to LazyVim's global LSP keymap list and applied **last** on `LspAttach` (`require("lazyvim.plugins.lsp.keymaps").set`). It is NOT inert. The snacks-picker extra (LazyVim default picker) maps `gd`‚Üí`Snacks.picker.lsp_definitions()` and `gr`‚Üí`Snacks.picker.lsp_references()`; appending `{ "gd", vim.lsp.buf.definition }` / `{ "gr", vim.lsp.buf.references }` therefore *wins* and reverts PHP nav to the raw quickfix-dumping handlers ‚Äî which is exactly the "gr does nothing" bug PR #2 introduced. Leave gd/gr/K unset so PHP uses the same picker nav as everything else. (`require("lazyvim.plugins.lsp.keymaps").get()` is deprecated in this version and warns.)
- **gd/gr/K resolution is intelephense indexing, not the keymap.** They resolve into project classes *and* `vendor/` only once intelephense has indexed a workspace root. `intelephense` root markers are `composer.json` / `.git` (`nvim-lspconfig/lsp/intelephense.lua`), both always present in a real Laravel project; `vendor/` must be populated (`composer install`) for vendor definitions to jump. Empty hover / "jumps nowhere" with a correct keymap means no root or not-yet-indexed, not a config bug.
- **Docblock noise is phpcs, NOT intelephense.** The "Missing doc comment for function ‚Ä¶" message is the phpcs `Squiz.Commenting.FunctionComment.Missing` sniff; intelephense's binary contains no such message. The `lang.php` extra enables phpcs via **nvim-lint** (`linters_by_ft.php = { "phpcs" }`). We disable it (`php = {}`) because the team standard is php-cs-fixer + IDE Helper, not enforced docblocks. Intelephense remains the diagnostics source with actionable errors on (undefined symbols/methods/types, type errors, argument count) and `unusedSymbols` off.
- **Formatting is php-cs-fixer** via conform.nvim (`formatters_by_ft.php = { "php_cs_fixer" }`, already set by the extra; we re-declare for clarity). conform's `php_cs_fixer` formatter prefers `vendor/bin/php-cs-fixer` and runs at the `composer.json` root, so a project `.php-cs-fixer.php`/`.dist.php` is auto-respected. Mason package name: `php-cs-fixer`.
- **Laravel IDE Helper:** `intelephense.files.maxSize` is raised to 5 MB so a large `_ide_helper_models.php` isn't skipped (default ~1 MB), and the helper files (`_ide_helper.php`, `_ide_helper_models.php`, `.phpstorm.meta.php`) are not excluded ‚Äî keeps model/facade completion working.
- **Laravel LSP is `laravel_ls`** (laravel-ls/laravel-ls; Mason package `laravel-ls`, lspconfig server `laravel_ls`), added alongside ‚Äî not replacing ‚Äî intelephense. intelephense = PHP semantics; laravel_ls = Laravel string-context (route/view/Blade/config/env/translation) intelephense lacks. lspconfig defaults: `cmd={"laravel-ls"}`, `filetypes={"php","blade"}`, `root_markers={"artisan"}` ‚Äî so it only attaches in real Laravel projects and stays quiet elsewhere. We keep `.blade.php` on the default `php` filetype (so PHP highlighting/treesitter still apply); laravel_ls covers blade files via the `php` filetype, no separate `blade` filetype registered. Pre-1.0 (alpha v0.1.0 as of 2026). `laravel-dev-tools` is archived (dead); `adalessa/laravel.nvim` is an Artisan plugin, not an LSP.
- **Mason org rename:** the plugin is `mason-org/mason.nvim`; the old `williamboman/mason.nvim` name causes a startup error.
- **Multi-client hover noise (noice):** noice (shipped on by LazyVim's `ui` spec, `event=VeryLazy`) overrides `vim.lsp.buf.hover` with a *per-client* handler (`noice/lsp/init.lua` ‚Üí `M.hover` uses `vim.lsp.buf_request`, and `on_hover` calls `vim.notify("No information available")` for each empty client). On multi-client buffers (Vue: vue_ls+vtsls+tailwindcss+eslint) every empty client pops a spurious notify even while another shows the real hover. Fix: disable noice's hover override (`opts.lsp.hover.enabled = false` in lsp.lua) so `K` falls back to **core** `vim.lsp.buf.hover`, which uses `buf_request_all` and only notifies when ALL clients are empty. noice's markdown overrides still style the core float; a genuine no-hover spot still reports once (verified headless: symbol-with-content ‚Üí 0 notifies, empty spot ‚Üí 1). Note: noice doesn't fully load in `--headless` (no VeryLazy), so reproduce the override from source, not headless.

## tmux
- Prefix is `C-Space`. Model: **sessions act as tabs** ‚Äî `prefix C-c` new session, `prefix s` picker, `prefix H`/`L` cycle, `prefix Space` last. Splits: `prefix |` (vertical), `prefix -` (horizontal).
- `detach-on-destroy off` is intentional (closing a session hops to another, not detach).
- Validate config parse: `tmux -L <sock> -f tmux/.tmux.conf new-session -d` then `kill-server`.
- **Status line is hand-rolled** in `tmux/.tmux.conf` (no status-line plugin ‚Äî `minimal-tmux-status` was dropped). Design: `status-position top`, transparent bar (`status-style bg=default` inherits ghostty's bg/transparency), rose-pine palette. Active window = gold (#f6c177) rounded pill (E0B6/E0B4 caps) w/ base (#191724) text; inactive = muted plain; session = iris (#c4a7e7) pill on the right after a subtle clock + prefix indicator. To restyle, edit `window-status-current-format`/`window-status-format`/`status-right` directly.
- **Verification gotcha:** tpm's `run '~/.tmux/plugins/tpm/tpm'` reads `~/.tmux.conf` (the live symlink), NOT the `-f` file you pass. So a `tmux -L test -f tmux/.tmux.conf` check loads whatever plugins the *installed* `~/.tmux.conf` declares and they override your hand-rolled options. To verify status-line options cleanly from a worktree, strip the tpm `run` line first: `grep -v "tpm/tpm'" tmux/.tmux.conf > /tmp/t.conf` then `tmux -L test -f /tmp/t.conf new-session -d` and `show-options -g`. In production `install` symlinks this file to `~/.tmux.conf`, so tpm sees the right declarations.
